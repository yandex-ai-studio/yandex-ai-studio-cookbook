# Best Practices: Как минимизировать галлюцинации AI-агента

Галлюцинации (генерация недостоверных фактов) — одна из главных проблем при создании автономных агентов. В этом разделе мы разберем проверенные подходы, которые помогут сделать ответы вашего агента мониторинга новостей максимально точными и фактичными.

Мы разделили методы на три уровня защиты: **Промпт-инжиниринг**, **Архитектура**, и **Валидация данных**.

---

## 1. Промпт-инжиниринг и системные инструкции

Правильно составленный `system_prompt` — это первая линия обороны.

### Explicit Constraints (Явные ограничения)
Всегда явно прописывайте, чего агент **не** должен делать. Модели лучше работают, когда границы жестко заданы.

### Citation Requirement (Требование цитирования)
Обязывайте агента ставить ссылки на источник после каждого утверждения. Это заставляет модель "заземлять" (grounding) свои ответы на конкретные URL.

**Паттерн:**
> "Дивиденды Газпрома составят 15 руб. [url: https://...]. Выручка упала на 10% [url: https://...]."

---

## 2. Улучшение работы с контекстом

Качество ответов напрямую зависит от качества найденной информации.

### Strict Domain Filtering (Фильтрация доменов)
Ограничение поиска только проверенными сайтами (white-listing) радикально снижает шум и фейки.
В вашем коде это уже сделано через `SEARCH_DOMAINS` — это **отличная практика**.

**Совет:** Разделите источники на "Tier 1" (официальные: сайт биржи, центр раскрытия информации) и "Tier 2" (новостные агрегаторы). При противоречиях отдавайте приоритет Tier 1.

### Recency Post-Filtering (Фильтрация по дате)
Модели часто путают старые новости с новыми, если в тексте сниппета нет явной даты.
**Решение:**
1. Передавайте текущую дату в системный промпт: `Сегодня {current_date}`.
2. Программно фильтруйте результаты поиска, если API возвращает дату публикации, отбрасывая всё, что старше N дней, *до* отправки в LLM.

### Low Temperature (Низкая температура)
Для задач фактологии (extraction/summarization) всегда используйте `temperature=0` или близкую к ней (0.1–0.3). Это снижает "креативность" модели и заставляет её держаться ближе к контексту.

## 3. Структурированный вывод и валидация (Pydantic)

Использование `Structured Output` (как `NewsDigest` в вашем коде) — один из самых эффективных методов борьбы с галлюцинациями структуры.

### Schema Validation
Используйте возможности Pydantic для проверки логики на лету.

**Пример валидатора:**
Если агент вернул оценку влияния `8/10`, но `sentiment` указан как "Нейтральное" — это логическая галлюцинация.

```python
from pydantic import BaseModel, validator

class Influence(BaseModel):
score: int
sentiment: str

    @validator('score')
    def check_score_sentiment_match(cls, v, values):
        sentiment = values.get('sentiment')
        if sentiment == 'Нейтральное' and v > 5:
            raise ValueError('Высокий score не может быть при нейтральном сентименте')
        return v
```
*Если валидация не проходит, можно автоматически попросить агента перегенерировать ответ с указанием ошибки.*

## 4. Self-Correction (Самопроверка)

Для критически важных дайджестов можно внедрить шаг "Рецензент" (Reviewer).

**Алгоритм:**
1. **Agent 1 (Searcher):** Ищет и пишет черновик.
2. **Agent 2 (Critic):** Получает черновик и исходные ссылки. Его задача — проверить, подтверждается ли каждое предложение ссылкой.
   - *Промпт Критика:* "Проверь, есть ли в тексте факты, не подтвержденные источниками. Если есть — удали их."

Это увеличивает время работы и стоимость, но гарантирует высокую точность для VIP-отчетов.
